## 新增需求

### 需求：TypeScript 严格模式

所有源代码必须使用启用严格模式的 TypeScript 编写。

#### 场景：严格模式配置

- **当** TypeScript 编译代码时
- **则** 必须在 tsconfig.json 中设置 `strict: true`

#### 场景：禁止隐式 any

- **当** 代码被编译时
- **则** 必须不允许隐式 `any` 类型

### 需求：基于接口的 props

所有组件 props 必须使用 TypeScript 接口定义，而不是内联类型。

#### 场景：Props 定义

- **当** 组件定义 props 时
- **则** 必须使用从 `types.ts` 导出的 `interface ComponentNameProps`

#### 场景：Props 使用

- **当** 组件使用 defineProps 时
- **则** 必须引用接口：`defineProps<ComponentNameProps>()`

### 需求：泛型组件类型

处理数据的组件必须支持泛型类型以实现类型推断。

#### 场景：泛型表格组件

- **当** CuiTable 与类型化数据一起使用时
- **则** 必须推断行类型：`<CuiTable<User> :data="users" />`

#### 场景：回调中的类型安全

- **当** 组件发出带有数据的事件时
- **则** 事件处理器必须接收正确类型的参数

### 需求：全局组件类型

组件库必须提供全局组件类型声明以支持 Volar IDE。

#### 场景：全局类型文件

- **当** 组件库被构建时
- **则** 必须包含带有 Vue runtime-core 模块增强的 `src/global.d.ts`

#### 场景：Volar 自动完成

- **当** 用户在模板中输入组件时
- **则** Volar 必须为所有组件库组件提供自动完成

#### 场景：Volar props 提示

- **当** 用户向组件添加 props 时
- **则** Volar 必须显示可用的 props 及其类型

### 需求：类型导出

所有公共类型必须从组件模块导出，并从主入口重新导出。

#### 场景：组件类型导出

- **当** 组件模块导出类型时
- **则** 必须导出所有 props 接口、事件类型和工具类型

#### 场景：主入口重新导出

- **当** 用户从包根目录导入类型时
- **则** 必须能够导入所有公共类型：`import type { CuiTableProps, CuiTableColumn } from 'vue-element-cui'`

### 需求：类型文档

复杂类型必须包含 JSDoc 注释以提供更好的 IDE 提示。

#### 场景：接口文档

- **当** 定义接口时
- **则** 必须包含描述用途和使用方法的 JSDoc 注释

#### 场景：属性文档

- **当** 定义接口属性时
- **则** 如果行为从名称上不明显，必须包含 JSDoc 注释
