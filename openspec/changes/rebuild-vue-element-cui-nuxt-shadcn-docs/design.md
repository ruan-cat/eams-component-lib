## Context

`vue-element-cui-nuxt` 的目标已经明确为 `vue-element-cui` 组件库官网型文档应用，而不是一个只承载 markdown 的通用文档皮肤。当前实现存在几类结构性问题：

- 文档站底座仍是旧方案，难以承载完整的组件库官网能力。
- 首页、导航、内容目录和组件页边界都围绕旧路径生长，无法支持新的组件类型分组。
- 示例展示体系过薄，无法稳定提供“先看效果，再看代码，再看说明”的 live demo 体验。
- 设计规范、最佳实践、开发约定等共性内容没有独立栏目，知识分散在旧组件页中。

本次变更已经在 `docs/plans/2026-03-13-vue-element-cui-nuxt-shadcn-docs-rebuild-plan.md` 中形成实施计划，并经本次会话确认以下关键边界：

- 以 `shadcn-docs-nuxt` 为底座。
- 目标是“组件库官网化”，而不是品牌营销站。
- 首页偏入口页，一级导航固定为“快速开始 / 组件 / 规范 / 更新”。
- 组件文档按组件类型分组，允许合并旧页面。
- live demo 由文档站内部手写维护，并直接运行真实组件。
- 旧文档路径全部废弃，不做兼容。
- 现有 markdown 文案尽量保留，以重编排代替重写作。

## Goals / Non-Goals

**Goals:**

- 用 `shadcn-docs-nuxt` 重建一个完整、可持续演进的组件库文档应用底座。
- 建立首页入口、一级导航、侧边导航和内容目录的一致信息架构。
- 建立文档站自有的 live demo 和内容组件体系，让核心组件页具备实时交互展示能力。
- 重组并迁移现有文档内容，使旧文案在新结构中尽量复用。
- 新增独立“规范”栏目，沉淀共性约定和最佳实践。
- 为文档站自有逻辑建立最小有效测试与验证路径。

**Non-Goals:**

- 不修改 `@eams-monorepo/vue-element-cui` 核心组件库的业务行为。
- 不保留任何旧文档路径的兼容层。
- 不扩展为品牌展示站、生态宣传站或路线图平台。
- 不优先建设复杂的外部 demo 复用系统。
- 不在本次变更中引入自动 API 文档生成流水线。

## Decisions

### 决策 1：以 `shadcn-docs-nuxt` 为底座，而不是继续在旧站点上增量修补

**选择**：直接把 `packages/vue-element-cui-nuxt` 重建为 `extends: ['shadcn-docs-nuxt']` 的文档应用。

**理由**：

- 用户已经明确要求按 `shadcn-docs-nuxt` 重做，而不是继续沿用旧主题。
- 旧站点的问题不是个别页面缺失，而是底座、导航、内容组织和展示层一起老化。
- 继续增量修补会把旧路径、旧结构和旧展示方式一起带进新站点，后续维护成本更高。

**替代方案**：

- 保留旧底座，只局部换主题：实现成本看似更低，但无法解决信息架构和 demo 体系的结构性问题。
- 直接复制其他模板站：会偏离“使用 `shadcn-docs-nuxt`”的主要求。

### 决策 2：采用“内容层 + 展示层”双层结构，而不是把所有能力塞进 markdown

**选择**：`content/` 承载文案内容，`app/` 与 `components/content/` 承载展示与交互能力。

**理由**：

- 用户要求尽量保留原有 markdown 内容，这意味着文案本身不应承担过多展示逻辑。
- 组件库官网需要统一的 demo 容器、API 表、说明块、入口组件，这些应由展示层负责。
- 分层后可以在不大改文案的前提下升级页面表现和交互能力。

**替代方案**：

- 仅靠 markdown + 少量 MDC 组件：短期可行，但无法稳定支撑完整的组件官网页面模型。

### 决策 3：组件文档按“组件类型”重组，允许合并旧页面

**选择**：组件栏目按类型组织，例如数据展示、表单输入、选择器、反馈与弹层、导航与布局、扩展能力。

**理由**：

- 这是本次会话中已经确认的一级组织方式。
- 旧页面边界存在明显重复，例如选择器相关页面天然适合合并为体系页。
- 类型分组更适合首页入口、组件索引、规范沉淀和后续扩展。

**替代方案**：

- 维持一页一个旧组件文档：迁移简单，但无法兑现“重大改造”和“官网化”的目标。
- 按业务场景组织：不符合本次确认的导航原则。

### 决策 4：live demo 由文档站手写维护，并直接运行真实组件

**选择**：在 `app/components/demo/` 中维护演示组件，并通过 `components/content/` 的包装组件接入文档页。

**理由**：

- 用户已经明确选择 demo 由文档站内部手写维护。
- 直接运行真实的 `@eams-monorepo/vue-element-cui` 组件，才能保证文档展示与实际行为一致。
- 这种结构能让“预览优先、代码可折叠、说明后置”的页面模型稳定复用。

**替代方案**：

- 复用旧示例或测试场景：约束太强，无法得到统一的官网展示质量。
- 只保留静态代码示例：不符合“实时交互演示优先”的要求。

### 决策 5：独立建立“规范”顶级栏目，而不是把约定散落到组件页

**选择**：新增 `content/3.guidelines/`，并收纳组件设计约定、表单与表格最佳实践、弹层交互建议、开发约定。

**理由**：

- 用户已明确要求“规范 / 最佳实践 / 开发约定”作为顶级栏目。
- 这些知识具有跨组件的共性，不适合被拆散在多个组件页中重复书写。
- 独立栏目能降低组件页冗余，也更方便后续新增约定。

**替代方案**：

- 完全嵌入组件页：页面会越来越重，且通用知识难以复用。

## Risks / Trade-offs

- [旧路径全部废弃会造成外部链接失效] → 这是已确认的破坏性决策；通过在 proposal、tasks 和最终交付说明中明确告知来管理风险。
- [现有内容迁移中可能出现文案遗漏或重组失真] → 迁移时坚持“能整段复用就整段复用”，并把内容核对列入最终验证任务。
- [真实组件样式接入失败会导致 live demo 失真] → 在 demo 体系完成前优先验证 `plugins/vue-element-cui.ts` 与样式加载链路。
- [完整官网化改造范围较大，容易在实现中失焦] → 以已确认的 5 个 capability 为边界，按 tasks 分组推进，不再临时扩 scope。
- [引入 `shadcn-docs-nuxt` 后仍可能需要少量自定义壳层] → 只在首页入口、内容组件和 demo 容器层扩展，避免深改底座内部实现。

## Migration Plan

1. 重写 `package.json`、`nuxt.config.ts`、`app.config.ts`、`app.vue` 等文档底座配置，完成 `shadcn-docs-nuxt` 接入。
2. 建立 `app/`、`components/content/`、`app/components/demo/` 等展示层目录，并先完成核心 demo 容器与基础测试。
3. 重排 `content/` 目录，先迁移首页、快速开始、更新栏目，再迁移组件栏目并合并同类页面。
4. 新建“规范”栏目和首页入口组件，补齐导航与入口体验。
5. 运行文档站测试、构建和关键页面手工核对，确认新的信息架构、live demo 和内容迁移全部成立。

**回滚策略：**

- 本变更集中在 `packages/vue-element-cui-nuxt/`，必要时可以整体回滚该目录和相关配置文件。
- 因为旧路径不会兼容保留，所以回滚只能通过 Git 回退整个变更，而不是局部切换路由。
