## 一句话概括

我在做 Vue 3 组件库迁移时，遇到过一次很典型的基础设施级事故：最初选用 `tsdown + unplugin-vue + rolldown` 构建组件库，但它在处理 `scoped SCSS` 的 Vue SFC 时出现了持续性构建失败。这个问题影响所有带 `scoped SCSS` 的组件，不是改一个业务组件就能解决的。我最后不是硬扛，而是通过分层排查确认这是工具链兼容性问题，然后果断切到 Vite library mode，把项目从不稳定的构建体系里拉出来。

## 面试时的 1 分钟回答版本

如果面试官问我，做这个组件库过程中最大的挑战是什么，我会说不是某个组件 API 设计，而是构建体系本身出过一次事故。

当时我为了让 Vue 3 组件库的构建更轻量，先选了 `tsdown`。它在 TypeScript 库构建上看起来很合适，但真正落到 Vue SFC，尤其是带 `scoped SCSS` 的组件后，构建开始持续报错。一类是 PostCSS 识别不了 SCSS 的 `//` 注释，另一类更严重，是 rolldown 在解析 Vue 样式虚拟模块时报 `Unexpected token`。这说明问题已经不是某个组件写法不规范，而是工具链组合本身不稳定。

我当时没有停留在“修一个报错”层面，而是把问题拆成语法层、样式层、SFC 编译层、bundler 兼容层逐步排查。最后判断继续深挖的收益已经很低，反而会拖慢整个组件库迁移，所以我及时止损，切到 Vue 官方生态更成熟的 Vite library mode。这个经历让我对工程化的理解更深了：基础设施选型里，成熟度和可验证性有时比理论上的速度优势更重要。

## 面试时的 3 到 5 分钟展开版本

## 背景

这个项目是把旧的 Vue 2 组件库迁到 Vue 3，并整理成 monorepo 里的现代组件库。组件本身要支持类型、样式隔离、后续文档站展示，所以构建工具不是一个边缘问题，而是整个项目的基础设施。

我最初选 `tsdown`，原因其实很合理：它配置轻、对 TypeScript 库构建友好、声明文件能力也不错。从“纸面能力”看，它比传统 Rollup 配置更省事。

## 真正的挑战是什么

挑战不在于“有报错”，而在于这个报错的性质很危险。

- 它不是单个组件有语法问题，而是所有带 `scoped SCSS` 的 Vue 组件都可能出问题。
- 它不是业务逻辑 bug，而是构建工具链层面的基础设施故障。
- 它会卡住整个组件库迁移，因为只要构建不稳定，后面所有组件开发都会变成在流沙上盖楼。

当时已经能看到两类信号：

- 一类错误是 PostCSS 不能正确处理 SCSS 注释。
- 另一类错误是更深层的 rolldown 对 Vue SFC 样式虚拟模块解析失败。

这两个信号放在一起，就说明不能再把它当成普通配置小问题看待了。

## 我是怎么排查的

我当时的思路不是到处试配置，而是先判断问题位于哪一层。

第一步，我先区分“表面问题”和“根本问题”。比如 SCSS 注释 `//` 报错，看上去可以通过改成 `/* */` 修掉，但这只能解决第一层错误。改完之后还有 `Unexpected token`，说明更深层的问题还在。

第二步，我把视角切到 Vue SFC 的构建链路。因为 Vue 的 `scoped` 样式本身会被编译成带查询参数的虚拟模块，如果 bundler 对这类模块支持不完整，就会出现“源码没问题，但工具链吃不下去”的现象。这个时候继续在组件代码上做微调，收益已经不高。

第三步，我做的是工程决策，而不是局部修补。我把可选方案都过了一遍：

- 继续改注释格式，只能治标。
- 去掉 `scoped`，会破坏样式隔离，不适合组件库。
- 升降级依赖，风险高，而且核心链路里 rolldown 还是 beta。
- 切到 Vite library mode，迁移成本可控，但稳定性和生态成熟度明显更高。

最后我选择了第四种，也就是切到 Vite。

## 为什么我认为这是正确决策

因为这件事本质上考验的不是“能不能硬修”，而是“能不能在项目推进中做对基础设施决策”。

我当时的判断标准主要有三个：

1. 这是组件库底座问题，不是局部功能问题，必须优先保证稳定。
2. Vue 生态里，Vite + 官方插件对 SFC、样式、类型生态的支持更成熟，风险更低。
3. 继续深挖 tsdown 的边界问题，会消耗项目节奏，沉没成本会越来越高。

所以我做的不是“放弃排查”，而是在完成根因判断之后及时止损。这个动作本身就是工程能力的一部分。

## 最终结果

后面我把构建体系迁到 Vite library mode，并补上对应的类型与样式构建能力，组件库恢复到可持续开发的状态。更重要的是，这次事故之后，我把几条经验沉淀下来了：

- 技术选型不能只看新和快，要看是否经过真实生产场景验证。
- 基础设施要尽早做全链路验证，尤其是组件库这种同时涉及类型、SFC、样式、打包的项目。
- 当问题已经指向工具链兼容性时，要敢于做替代方案决策，而不是被沉没成本绑住。

如果面试官继续追问，我会把这次经历定义为：一次从构建事故里反推出工程方法论的过程。

## 面试官可能追问的问题与回答思路

**Q：你为什么一开始会选 tsdown？**

A：因为从目标看它并不是错误选择。它对 TypeScript 库构建友好、配置轻、声明生成方便，纸面上很适合做组件库。但这次让我学到，组件库不是纯 TS 库，Vue SFC、`scoped SCSS`、样式虚拟模块这些链路会显著放大工具链兼容性问题。

**Q：你为什么不继续修，而是直接迁移？**

A：因为我判断那已经不是单点配置问题，而是工具链组合的边界问题。继续修的成本不确定，而且会拖住整个迁移节奏。对于基础设施，稳定可交付比局部最优更重要。

**Q：这件事之后你会怎么避免再发生？**

A：我会在项目早期就做全链路样板验证，不只验证“能不能打包”，还要验证 Vue SFC、类型、样式、`scoped`、产物结构是否都能稳定工作。另外在核心基础设施上，我会优先选择官方生态里经过验证的方案。

**Q：这件事最能体现你的什么能力？**

A：我觉得是两点。一是分层定位问题的能力，不会被表面报错带偏。二是工程决策能力，知道什么时候继续优化，什么时候应该止损换轨。
